# 多终端实例与复制终端设计

## 背景

当前会话连接流程使用 `terminal-${sessionId}` 作为前端标签 ID。后端虽然每次连接都会创建新的 SSH 连接和 PTY，但前端 Store 会把相同会话的后续连接识别为已有标签，导致一个会话只能显示一个终端实例。

本设计允许同一会话同时创建多个独立终端，并在已连接终端标签的右键菜单中提供“复制终端”。这里的复制是使用相同会话配置发起新的 SSH 登录，不是复制数据库会话记录，也不克隆远端 Shell 进程。

## 目标

1. 每次双击会话或点击“连接”都创建新的 SSH 连接、PTY 和顶部终端标签。
2. 同一会话的标签按 `会话名`、`会话名 #2`、`会话名 #3` 的方式区分。
3. 已连接终端标签的右键菜单提供“复制终端”。
4. 复制出的终端与原终端独立维护输入、输出、录制、SFTP、分屏、连接状态和关闭流程。
5. 连接失败时不创建标签，也不占用实例编号。

## 非目标

- 不复制当前 Shell 工作目录、进程、环境变量、屏幕历史或滚动位置。
- 不复制数据库中的会话记录。
- 不改变后端 SSH、PTY、认证和终端关闭接口。
- 不持久化已打开终端或实例编号。

## 架构

新增一个纯前端终端标签辅助模块，集中负责：

- 根据后端返回的唯一 `terminalId` 生成标签 ID。
- 根据同一 `sessionId` 当前打开的终端标签选择最小可用实例编号。
- 根据实例编号生成可读标题。

`useSession.connect(sessionId)` 继续作为唯一建连入口。普通会话连接和标签右键“复制终端”都调用该方法，从而共享认证弹框、后端 `TerminalService.Open`、连接状态、最近连接刷新、日志和错误处理。

## 标签模型

终端标签增加可选字段：

```ts
interface Tab {
  id: string
  title: string
  type: 'terminal' | 'playback'
  terminalId?: string
  sessionId?: number
  terminalInstance?: number
}
```

新终端标签遵循：

```ts
{
  id: `terminal-${terminalId}`,
  title: terminalInstance === 1 ? session.name : `${session.name} #${terminalInstance}`,
  type: 'terminal',
  terminalId,
  sessionId: Number(sessionId),
  terminalInstance,
}
```

实例编号只根据当前打开且 `type === 'terminal'`、`sessionId` 相同的标签计算。选择从 1 开始的最小未使用正整数，既避免重复标题，也允许关闭中间标签后复用编号。连接失败不会写入 Store，因此不会占用编号。

## 交互

### 普通连接

1. 用户双击会话或点击“连接”。
2. 显示现有连接进度弹框。
3. 后端创建新的 SSH 连接和 PTY，返回唯一 `terminalId`。
4. 前端计算实例编号并创建唯一标签。
5. 新标签成为当前活动页面并获得终端焦点。

即使已有同会话终端，步骤 3 也始终执行，不跳转或复用旧标签。

### 复制终端

1. 用户右键一个已连接终端的顶部标签。
2. 菜单显示带复制图标的“复制终端”。
3. 点击后使用该标签的 `sessionId` 调用统一建连入口。
4. 新连接成功后创建下一个可用编号的终端标签。

回放标签不显示“复制终端”。缺失 `sessionId` 的遗留终端标签禁用该操作，避免发起无效请求。

## 错误处理

- 后端建连失败沿用连接弹框错误状态，不创建标签。
- 缺失会话记录时 `connect` 直接返回，不调用后端。
- 缺失 `sessionId` 的终端标签不允许复制。
- 右键菜单只负责触发统一入口，不自行维护加载状态或错误提示。

## SFTP 实例隔离

- 每个终端标签使用自身 `tab.id` 保存 SFTP 面板打开状态。
- 切换到其他终端或工作区时，已打开的 SFTP 面板随终端隐藏层保持挂载，返回后保留该实例的路径和列表状态。
- 文件拖放目标使用 `sftp-drop-zone-${terminalId}`，不能只使用共享的 `sessionId`。
- 拖放事件必须精确匹配目标 `terminalId`，缺失或不匹配目标 ID 时不执行上传。
- 关闭某个终端标签时清理该标签对应的 SFTP 打开状态，不影响同会话其他实例。

## 测试策略

### 纯函数测试

- 首次连接生成实例 1 和基础标题。
- 连续连接生成实例 2、3。
- 中间编号关闭后复用最小可用编号。
- 不同会话和回放标签不参与编号。
- 标签 ID 使用后端 `terminalId`，不使用 `sessionId`。

### Hook 行为测试

- 同一会话连续连接两次调用后端两次并创建两个标签。
- 两个标签拥有不同 ID、terminal ID 和正确标题。
- 连接失败不增加标签。

### 组件交互测试

- 右键终端标签可见“复制终端”。
- 点击菜单项以原标签 `sessionId` 调用 `connect`。
- 回放标签不显示复制操作。
- 同会话两个终端的 SFTP 面板打开状态互不继承，切换后各自保持。
- 同会话两个终端使用不同的 SFTP drop target ID。
- 实际 `App -> SessionWorkspaceProvider -> WindowTitleBar` 组件树能够把统一 `connect` 提供给复制操作。

## 验收标准

1. 同一会话连续连接三次可同时操作三个独立终端。
2. 标签标题依次为基础名称、`#2`、`#3`，且标签 ID 不冲突。
3. 右键任意已连接终端并点击“复制终端”会建立新的 SSH 登录。
4. 关闭、录制、SFTP 和分屏只影响各自终端实例。
5. 连接失败不产生空标签或跳号。
6. 新增行为具备自动化测试，并通过前端覆盖率、Go 测试、lint 和 `wails3 build`。
