# 顶部一级终端标签设计

## 背景

当前 SSH 终端和录制回放位于“会话”工作区内部。建立连接后，终端持续占据会话资产页面右侧，用户再次点击顶部“会话”无法返回资产管理页面。固定工作区与动态终端标签使用两套激活状态，也容易产生页面选择不一致。

本设计将会话、宏、SSH 终端和录制回放统一提升为窗口顶部的一级页面，并让顶部固定导航与左侧边栏支持联动收缩。

## 目标

1. 点击顶部“会话”始终能够展示会话资产页面。
2. 切换工作区时保持后台 SSH 连接和终端渲染状态。
3. 会话、宏、SSH 终端和回放使用统一的一级导航模型。
4. 顶部固定菜单与左侧边栏由同一个三条杠按钮同步收缩和展开。
5. 多终端切换时保持光标、尺寸、焦点和 addon 生命周期隔离。
6. 单个终端异常不能导致整个应用黑屏。

## 非目标

- 应用重启后不自动恢复 SSH 连接或录制回放。
- 不改变 SSH 建连、认证、录制文件格式和 SFTP 业务流程。
- 不重新设计会话资产页、宏页面和终端主题。
- 不持久化当前打开的动态终端标签。

## 视觉与布局

采用单行统一标签栏方案。

顶部从左到右排列：

1. 三条杠菜单按钮。
2. 固定“会话”标签。
3. 固定“宏”标签。
4. SSH 终端和录制回放动态标签。
5. 动态标签快捷列表。
6. 可拖动空白区域。
7. 主题、设置和窗口控制按钮。

“会话”和“宏”始终位于动态标签之前。动态标签区域空间不足时允许横向滚动，并通过快捷列表定位任意标签。窗口控制和应用操作始终固定在右侧。

所有交互控件必须位于 Wails 的 no-drag 区域；只有明确的空白区域可作为窗口拖动区域。

## 联动收缩

三条杠按钮统一控制顶部固定菜单和左侧边栏。

### 展开状态

- 显示顶部“会话”和“宏”。
- 显示当前工作区对应的左侧边栏。
- 恢复用户上次调整的侧边栏宽度。

### 收缩状态

- 隐藏顶部“会话”和“宏”。
- 隐藏左侧边栏及其边框。
- 保留三条杠、动态终端、回放和窗口控制。
- 不改变当前一级页面。
- 不关闭 SSH、不停止回放、不卸载终端组件。

当当前页面为会话或宏时，收缩仅释放左侧空间，主工作区内容继续显示。再次展开后恢复原工作区、侧边栏宽度和可恢复的滚动位置。

## 状态模型

使用判别联合统一表示当前一级页面：

```ts
type ActiveSurface =
  | { type: 'workspace'; id: 'sessions' | 'macros' }
  | { type: 'terminal'; id: string }
  | { type: 'playback'; id: string }
```

应用状态至少包含：

```ts
type NavigationState = {
  activeSurface: ActiveSurface | null
  navigationCollapsed: boolean
  sidebarWidth: number
}
```

`activeSurface` 是唯一的页面激活来源。`null` 只表示应用启动后的 Welcome 页面；用户首次进入任意工作区或动态标签后，本次运行期间不再返回 Welcome。原有 `sidebarTab` 和 `activeTabId` 不再独立决定页面内容，避免固定工作区与动态标签同时处于激活状态。

`navigationCollapsed` 同时驱动顶部固定菜单和左侧边栏，不允许组件各自维护收缩状态。

## 组件边界

### UnifiedTitleBar

- 渲染三条杠菜单、固定工作区、动态标签和窗口控制。
- 只消费导航状态并触发 store action。
- 不直接管理 SSH、回放或 xterm 生命周期。

### DynamicTabStrip

- 渲染 SSH 终端和回放标签。
- 负责横向滚动、活动标签可见性和快捷列表。
- 根据标签类型分派关闭动作。

### WorkspaceLayer

- 根据 `activeSurface` 显示 Welcome、会话资产或宏工作区。
- Welcome 只在应用启动且用户尚未进入任何一级页面时显示。
- 工作区切换不影响动态终端挂载状态。

### TerminalLayer

- 为每个 SSH 标签保持独立的终端组件实例。
- 非活动终端通过隐藏容器保留挂载，不使用条件渲染销毁。
- 终端恢复可见后执行 fit、refresh 和 focus。
- 每个终端独立持有 addon、尺寸观察器和清理函数。

### PlaybackLayer

- 与 SSH 终端使用相同的一级页面切换方式。
- 播放器生命周期与后端 SSH terminal ID 完全分离。
- 关闭回放仅停止本地播放并移除回放标签。

### TerminalErrorBoundary

- 每个动态终端页面单独包裹 Error Boundary。
- 捕获渲染和生命周期异常，展示可关闭、可重试的错误卡片。
- 单个终端失败不能卸载其他终端或应用根节点。

## 交互流程

### 返回会话资产页

1. 用户点击顶部“会话”。
2. store 将 `activeSurface` 设置为 `workspace/sessions`。
3. 会话资产页显示。
4. SSH 和回放组件继续挂载，后台数据继续处理。

### 激活终端

1. 用户点击 SSH 动态标签。
2. store 将 `activeSurface` 设置为对应 terminal ID。
3. 终端容器变为可见。
4. 下一渲染帧依次执行 fit、refresh 和 focus。
5. 标签条滚动到活动标签可见位置。

### 关闭 SSH 终端

1. 已连接终端先显示关闭确认。
2. 用户确认后调用后端关闭接口。
3. 后端关闭成功或明确返回终端已不存在后，移除前端标签。
4. 关闭当前标签时优先激活右侧相邻标签，其次左侧相邻标签。
5. 没有动态标签时返回会话资产页。

后端关闭失败时保留标签并显示错误，不允许静默丢失仍可能活动的连接。

### 关闭回放

1. 停止计时器和本地播放器。
2. 清理回放终端实例。
3. 移除回放标签。
4. 按相同的相邻标签规则选择下一页面。
5. 不调用 SSH 终端关闭接口。

### 连接断开

- 保留终端标签和历史输出。
- 状态灯显示红色断开状态。
- 提供重新连接和关闭操作。
- 断开事件不会自动切换当前页面。

## 终端可见性恢复

终端从隐藏状态恢复时，必须等待容器具有非零尺寸后再调整：

1. 将目标容器设为可见。
2. 使用下一次 `requestAnimationFrame` 等待布局完成。
3. 调用 FitAddon 的 `fit()`。
4. 调用终端 `refresh(0, rows - 1)`。
5. 仅在用户主动点击终端标签时调用 `focus()`。

组件卸载清理必须幂等。React 开发模式重复执行 effect 时，不得重复释放同一个 addon、observer 或 terminal 实例。

## 状态持久化

- 持久化 `navigationCollapsed`。
- 继续持久化有效范围内的 `sidebarWidth`。
- 不持久化 `activeSurface` 中的 terminal 或 playback ID。
- 应用启动仍显示 Welcome；用户点击会话或宏后 Welcome 不再出现。

## 可访问性

- 三条杠按钮提供明确的 `aria-label` 和展开状态 `aria-expanded`。
- 所有顶部标签支持键盘聚焦和 Enter/Space 激活。
- 关闭按钮具有包含标签名称的辅助文本。
- 活动标签使用 `aria-selected`，标签容器使用正确的 tab 语义。
- 状态灯不能只依赖颜色，必须同时提供文本或辅助描述。

## 测试策略

### Store 单元测试

- 工作区、终端和回放之间切换时只有一个活动页面。
- 收缩状态同步影响顶部固定菜单与左侧边栏。
- 关闭当前、非当前和最后一个动态标签时选择正确页面。
- 展开后恢复有效侧边栏宽度。

### 组件交互测试

- 点击会话后资产页重新出现，SSH 标签仍存在。
- 点击三条杠后顶部固定菜单与左侧边栏同步隐藏和恢复。
- 动态标签支持激活、关闭、横向滚动和快捷列表定位。
- 关闭连接中的 SSH 标签需要确认，关闭回放不调用 SSH 接口。
- 键盘操作和 ARIA 状态符合设计。

### 终端生命周期测试

- 切换工作区不会调用 `terminal.dispose()`。
- 终端恢复后按 fit、refresh、focus 顺序执行。
- 非主动切换导致的恢复不抢占用户焦点。
- 多终端的 addon、尺寸观察器和清理函数相互隔离。
- 重复清理不会再次释放已释放资源。

### 容错测试

- 单个终端抛出异常时 Error Boundary 接管，应用其他区域可继续操作。
- SSH 断开后保留历史输出和断开状态。
- 后端关闭失败时标签保留并显示错误。
- 回放关闭不会将录制文件路径作为 terminal ID 传给后端。

## 验收标准

1. 登录 SSH 后点击“会话”，会话资产页立即显示且 SSH 连接保持。
2. 点击任意动态终端标签可恢复正确终端、尺寸和光标位置。
3. 三条杠能同步收缩或展开顶部固定菜单和左侧边栏。
4. 收缩状态下动态标签、主题、设置和窗口按钮仍可操作。
5. 多个 SSH 终端之间切换不共享焦点、光标或 addon 生命周期。
6. 单个终端异常不会造成整个应用黑屏。
7. 关闭 SSH、关闭回放和连接断开遵循各自独立流程。
8. 所有新增行为有自动化测试看护，并通过项目既有质量门禁。

## 验证门禁

实施完成后必须执行：

- 前端单元测试、覆盖率、lint 和生产构建。
- Go 单元测试和 race 检测。
- `goimports-reviser` 和 `golangci-lint`。
- `wails3 build`。

全部验证通过后方可提交并推送实现代码。
